## 1. 项目背景

### 1.1 问题陈述

求职过程中，应聘者通常会同时关注多家公司的机会。**通勤距离与时间是决定 offer 接受度的关键因素之一**，但目前缺少一个工具能：

- 基于真实地图数据，**批量计算多家公司的通勤情况**
- 模拟**真实的工作日通勤场景**（早高峰出发、晚高峰返程）
- 在地图上**直观对比**多家公司的位置与通勤成本
- 帮助求职者**快速筛选**地理位置合适的目标公司

### 1.2 目标用户

| 用户类型 | 特征 | 核心诉求 |
|---------|------|---------|
| 主动求职者 | 正在投简历、面试 | 评估目标公司通勤可行性 |
| 跳槽考虑者 | 已在职但有意向 | 横向对比多家公司通勤成本 |
| 租房决策者 | 即将搬家 | 用多个候选住址 + 公司组合做选址决策 |

### 1.3 产品定位

**一句话定位**：求职者的通勤成本可视化决策工具。

**非目标**（明确不做的事）：
- ❌ 不做综合招聘平台（不抓取职位信息）
- ❌ 不做社区/点评（不评价公司好坏）
- ❌ 不做实时导航（仅做规划层面的通勤估算）

## 2. 产品概述

### 2.1 产品形态

- **Web 应用**（PC + 移动端响应式 H5）
- 后续可能扩展：微信小程序

### 2.2 核心价值主张

> 输入「我家在哪」+「我想去哪个城市做什么岗位」→ AI 推荐潜在公司 → 一键看到所有公司的通勤地图与时间对比。

### 2.3 用户旅程（Happy Path）

```
1. 打开应用
   ↓
2. 设置家庭住址（地图选点 / 地址搜索）
   ↓
3. 输入所在城市 + 求职岗位
   ↓
4. AI 推荐当地相关公司列表
   ↓
5. 勾选感兴趣的公司
   ↓
6. 查看地图标注 + 通勤时间表
   ↓
7. 切换出行方式 / 调整时间
   ↓
8. 导出对比报告 / 保存历史记录
```

### M1. 个人信息管理

#### 功能描述
保存用户的基础信息，作为后续推荐和计算的输入。

#### 字段清单
| 字段 | 类型 | 必填 | 备注 |
|------|------|------|------|
| 当前所在城市 | 字符串 | 是 | 下拉选择主流城市 + 支持搜索 |
| 求职岗位 | 字符串 | 是 | 自由输入，如"后台开发""产品经理" |
| 工作经验年限 | 数字 | 否 | 0-30，用于 AI 更精准推荐 |
| 期望公司类型 | 多选 | 否 | 大厂 / 中厂 / 创业公司 / 外企 |

#### 交互细节
- 首次进入应用时引导填写
- 信息可随时在"设置"页面修改
- 修改后影响后续 AI 推荐结果（不影响已勾选公司）

### M2. 家庭住址管理

#### 功能描述
支持用户保存**多个候选住址**，便于租房选址对比。

#### 功能点
- ✅ 新增住址（地址搜索 / 地图点选）
- ✅ 编辑住址（修改名称、坐标）
- ✅ 删除住址
- ✅ 标记"当前默认住址"
- ✅ 给住址打标签（如"现居""候选A""候选B"）

#### 数据字段
| 字段 | 类型 | 必填 | 备注 |
|------|------|------|------|
| 住址别名 | 字符串 | 是 | 用户自定义，如"我家""朝阳房源" |
| 详细地址 | 字符串 | 是 | 高德地理编码返回的标准地址 |
| 经度 | 浮点 | 是 | 高德坐标系 (GCJ-02) |
| 纬度 | 浮点 | 是 | 高德坐标系 (GCJ-02) |
| 是否默认 | 布尔 | 是 | 同一时刻只有一个默认 |
| 备注 | 文本 | 否 | 用户备注，如"月租 5000" |

#### 交互细节
- 默认住址在首页地图上以**蓝色标记**显示
- 切换默认住址后，所有通勤计算自动以新住址为起点

### M3. AI 公司推荐

#### 功能描述
基于用户的"城市 + 岗位"，调用豆包 AI 生成本地相关公司列表。

#### 输入参数
- 城市（如"北京"）
- 岗位（如"后台开发"）
- （可选）公司类型偏好
- （可选）经验年限

#### 输出结构
AI 返回 JSON 格式公司列表：

```json
{
  "companies": [
    {
      "name": "字节跳动",
      "category": "大厂",
      "industry": "互联网",
      "address_hint": "北京市海淀区中关村",
      "reason": "国内顶级互联网大厂，后台开发岗位需求量大"
    }
  ]
}
```

#### 数量与分级
- 每次推荐 **15-30 家**公司
- 按分级展示：
  - **大厂**（如 BAT、字节、美团）
  - **中厂**（如知名独角兽、上市公司）
  - **创业公司**（融资中或新兴）

#### 交互细节
- 用户点击"AI 推荐"按钮触发
- Loading 状态显示"AI 思考中..."
- 推荐结果支持**刷新**（重新生成）
- 推荐结果支持**追加**（生成更多）
- 每家公司显示：名称、类型、行业、推荐理由、勾选按钮

#### 注意事项
- AI 返回的地址是"提示"，需通过高德地理编码获取精确坐标
- 若 AI 推荐的公司在高德搜不到，标记"地址待补充"，允许用户手动定位

### M5. 通勤计算与展示

#### 功能描述
**核心功能**。基于高德路径规划 API，计算住址 ↔ 公司的通勤情况。

#### 计算维度

##### 时间策略
| 模式 | 出发地 | 出发时间 | 到达地 | 用途 |
|------|--------|---------|--------|------|
| **早通勤** | 家 | 周一 08:00（默认，可改） | 公司 | 上班通勤 |
| **晚通勤** | 公司 | 周一 17:30（默认，可改） | 家 | 下班通勤 |
| **逆推模式** | 家 / 公司 | 反推 | 自定义到达时间 | "我想 9:00 到公司，几点出门" |

##### 出行方式
用户**多选**，可同时展示多种方式的对比：
- 🚇 公交/地铁
- 🚗 驾车
- 🚴 骑行
- 🚶 步行

##### 输出字段
每个 [住址 × 公司 × 出行方式 × 时间段] 组合返回：
| 字段 | 说明 |
|------|------|
| 总耗时 | 分钟数 |
| 总距离 | 公里 |
| 出发时间 | HH:MM |
| 到达时间 | HH:MM |
| 路线详情 | 换乘次数、各段距离/时间 |
| 费用估算 | 公交票价/打车费用（高德返回） |

#### 列表展示
表格视图，列字段：
| 公司 | 距离 | 早通勤 | 晚通勤 | 出行方式 | 操作 |
|------|------|--------|--------|---------|------|
| 字节跳动 | 12.3 km | 公交 55 min | 公交 62 min | 🚇 公交 | 详情 / 移除 |

支持：
- 按耗时排序
- 按距离排序
- 按出行方式筛选
- 点击"详情"查看路线分段

#### 详情面板
点击某条记录展开：
- 路线地图（高亮显示）
- 分段说明（如"步行 5 min → 地铁 6 号线 8 站 → 步行 3 min"）
- 出行小贴士（如换乘提醒）

### M8. 多公司对比视图（P1）

#### 功能描述
横向对比多家公司的通勤数据，可视化决策。

#### 视图类型
1. **柱状图**：X 轴=公司，Y 轴=通勤时间（早/晚双柱）
2. **散点图**：X 轴=距离，Y 轴=时间
3. **雷达图**：维度=[距离, 早通勤, 晚通勤, 换乘次数, 费用]，多公司叠加对比
4. **热力表**：行=家庭住址，列=公司，单元格=通勤时间（适合多住址对比）

## 4. 非功能性需求

### 4.1 性能
| 指标 | 目标 |
|------|------|
| 首屏加载 | < 2 秒 |
| AI 推荐响应 | < 10 秒（含 Loading 友好提示） |
| 单次通勤计算 | < 3 秒 |
| 地图加载 | < 2 秒 |

### 4.2 兼容性
- **浏览器**：Chrome / Safari / Edge 最新两个版本
- **移动端**：iOS 14+ Safari，Android 10+ Chrome
- **屏幕**：最小 320px 宽度

### 4.3 安全
- 高德、豆包 API Key 仅存储在后端 `.env`
- 前端调用 AI 必须经过后端代理（防止 Key 泄露）
- 用户数据本地化存储（不上传第三方）

### 4.4 可维护性
- 后端模块化（handler / service / repository 分层）
- 前端组件化（单文件 < 300 行）
- Docker 化一键启动
- 完整中文注释与文档

### 4.5 扩展性
- 数据库 Schema 预留 `user_id`、`tenant_id`
- AI 提供商可配置（豆包 / OpenAI / DeepSeek 可切换）
- 地图提供商可配置（高德 / 百度 / 腾讯）

## 7. 关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 高德 API 配额超限 | 通勤计算失败 | 添加结果缓存（同参数 24h 内复用） |
| AI 返回的公司不真实 | 推荐质量差 | Prompt 中强制要求"真实存在的公司"；前端给 Disclaimer |
| AI 返回地址不准确 | 定位错误 | 用高德 POI 搜索做二次校验 |
| 周一时间预估不准 | 结果偏差 | 文案标注"基于高德历史路况估算" |
| 地址搜索结果歧义 | 选错位置 | 强制用户从下拉列表确认 |

## 10. 待澄清问题（已决议）

> 决议日期：2026-04-15

| # | 问题 | 决议 | 备注 |
|---|------|------|------|
| 1 | AI 推荐结果是否缓存？ | ✅ 缓存 | 同 (city + position) 输入 **24 小时内**复用，过期重新调用豆包；强制刷新接口可绕过缓存 |
| 2 | 高德 API 调用方案？ | ✅ 混合 | **前端 JS SDK** 负责地图渲染、用户交互（地址搜索、点选）；**后端 Web Service** 负责路径规划、地理编码、批量计算（保护 Key） |
